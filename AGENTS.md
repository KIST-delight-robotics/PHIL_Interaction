# `phil-brain` (phil_robot) 작업 지침

이 레포는 필(Phil)의 **LLM brain** 이다. Whisper STT, Ollama 기반 classifier/planner,
명령 검증, MeloTTS를 묶은 Python 판단 계층이며, 검증된 명령을 `phil-controller`로 보낸다.

## 시스템 안에서의 위치

```text
[phil-brain (LLM)]  ──TCP 9999──▶  phil-controller (C++)  ──CAN/DXL──▶  phil-sil 또는 실제 하드웨어
```

이 레포는 **Contract A**(brain ↔ controller, TCP 9999)의 client 측 구현이다.
계약 전문은 [CONTRACTS.md](./CONTRACTS.md)에 있다. TCP client는 `runtime/phil_client.py`,
명령 문법/관절 범위 검증은 `pipeline/command_validator.py`에 있다.

> **계약을 바꾸지 마라 — 합의 없이는.** 보내는 명령 문법(`move:`/`gesture:`/`p:` 등)과 읽는 상태 JSON
> 필드는 `phil-controller`의 `AgentSocket`와 일치해야 한다. 바꿔야 하면 먼저 `CONTRACTS.md`를 고치고
> 세 레포 복사본을 동기화한 뒤 `phil-controller`도 함께 수정한다.

## 페르소나 규칙

이 레포에서는 필(Phil)의 대화/행동 페르소나가 여러 prompt와 shortcut 응답에 걸쳐 반복되므로, 아래 규칙을 공통 기준으로 사용한다.

## 페르소나 작성 규칙
- 페르소나는 항상 `고정 정체성`, `상호작용 톤`, `능력/제약`을 구분해서 작성한다.
- `고정 정체성`에는 검증된 사실만 넣는다.
  - 기본값: 필(Phil), KIST에서 개발된 지능형 휴머노이드 드럼 로봇
- `능력`은 실제 코드/스킬/상태 기반으로 뒷받침되는 것만 적는다.
  - 지원하지 않는 곡, 동작, 센서, 추론 능력을 페르소나 문구로 먼저 약속하지 않는다.
- `제약`은 숨기지 말고 명시한다.
  - 예: 안전 키 미해제, 연주 중 동작 제한, 관절 한계, 현재 상태 의존 동작
- 확인형 정체성 질문에 대한 yes/no는 흔들리지 않게 쓴다.
  - 필이 맞는 질문은 긍정, 아닌 이름/정체성은 부정으로 일관되게 응답한다.
- 말투는 친절하고 짧게 유지하되, 과장된 감정 연기나 불필요한 설정을 덧붙이지 않는다.
- 새 페르소나 규칙을 추가할 때는 특정 한 문장 예시보다 재사용 가능한 패턴으로 적는다.
  - 예: `모펫이니? -> 아니요` 같은 단일 예시만 적지 말고, `이름 확인형 부정 질문은 부정 응답과 부정 제스처를 우선한다`처럼 적는다.
- prompt 안의 페르소나 문구는 가능한 한 짧게 유지한다.
  - 반복되는 장식 문구보다 정체성/제약/행동 우선순위를 먼저 적는다.

## 코드 규칙
- 변수명은 3단어 이하. 약어는 언더바로 분리(`op_cmd`, `joint_deg`). 약한 이름(`it`, `iter`)보다 역할이
  드러나는 이름을 쓴다.
- 제어 판단 로직은 `pipeline/`, 하드웨어/오디오/네트워크 런타임 결합은 `runtime/`, 오프라인 평가는 `eval/`에 둔다.
- 엔트리포인트(`phil_brain.py`)는 얇게 유지하고, 제어 로직은 `pipeline/` 안에서만 확장한다.

## 작업 로그
- 이 레포에서 의미 있는 변경을 하면 레포 루트의 `log.md`에 기록한다.
- 각 항목에 날짜(`YYYY-MM-DD`), 시간(`HH:MM KST (UTC+9)`), 변경 요약을 포함하고, 최신 항목을 위에 추가한다.

## 우선순위
- 이 파일은 `phil-brain`(phil_robot) 전체에 적용된다.
- `eval/AGENTS.md`처럼 더 하위 경로의 지침이 있으면 그 파일이 우선한다.
- 사용자/시스템/개발자 지침이 있으면 최우선으로 따른다.
